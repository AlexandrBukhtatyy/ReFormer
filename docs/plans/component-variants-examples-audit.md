# Аудит компонентов: Варианты и Примеры

Для каждого компонента: **Варианты** = собранные пресеты (комбинации настроек, БЕЗ стилизации), **Примеры** = приёмы использования возможностей. Составлено суб-агентами по исходникам ui-kit / cdk и текущим doc-конфигам.

## Свод правил (как проводить границу)

- disabled → ВСЕГДА Вариант. Это настройка-состояние собранного компонента. Механизм задания (native prop, control.disable(), авто-наследование от disabled-массива) на классификацию не влияет. Подтверждено единообразно: Input, InputPassword, InputMask, Textarea, Checkbox, RadioGroup, Select, Button, FormField, FormArray.
- invalid / aria-invalid (состояние ошибки) → ВСЕГДА Вариант. Красная рамка/ring — это состояние, а не стилизация, даже если визуально выражается CSS-классами (aria-invalid:border-destructive). Механизм срабатывания (провал валидатора) идёт в Пример, но сам показ invalid-состояния — Вариант.
- value пусто / заполнено → ВСЕГДА Вариант. Наличие значения — базовая ось витрины состояний для всех form-bound контролов.
- placeholder (наличие/кастомный текст) → Вариант. Это содержательная настройка подсказки, НЕ стилизация. Для InputMask особый случай: placeholder по умолчанию = mask, а переопределение placeholder — отдельный Вариант.
- type / режимные дискриминанты → Вариант. Input type (text/email/tel/url/password/number), количество опций и наличие групп (Select, RadioGroup), Button type (button/submit/reset), содержательная композиция (Button text/icon/icon-only), Textarea native maxLength, titleAs (Section), defaultOpen (Collapsible), позиция навигации (FormWizard первый/промежуточный/последний шаг) — всё это настройки-режимы уже собранного компонента.
- Валидаторы (required(), email(), minLength/maxLength как бизнес-правило, pattern) → ВСЕГДА Пример. Это возможность интеграции с @reformer/core, а не состояние. Показ проваленной валидации (invalid) — Вариант, но подключение валидатора — Пример. Единообразно: Input, InputPassword, InputMask, Textarea, Checkbox, RadioGroup, Select, ErrorState.
- onChange / onBlur / binding-wiring → НИКОГДА не Вариант. Это провод (control API), задействуется только в Примерах (реактивное чтение, счётчик символов, нормализация на blur, привязка к model.$).
- Opt-in возможности (фичи, которые надо ПОКАЗАТЬ в коде) → ВСЕГДА Пример: clearable, reveal/переключение видимости пароля, галерея mask-паттернов, resource/data-provider со стратегиями, asChild/Slot, imperative ref-handle, computed/onChange-поведение (зависимая блокировка), кастомный render (Count/ItemIndex/Error), useFormControl/useFormField/useFormArray hooks, композиция под-компонентов.
- Стилизация исключается из Вариантов ПОЛНОСТЬЮ: className, size/width/height, цвет, тема, отступы, rows (Textarea), Button variant + size, cardClassName, titleClassName/contentClassName. Осей-вариантов по размеру/цвету/теме не создаём никогда. ЕДИНСТВЕННОЕ исключение — ориентация раскладки (RadioGroup horizontal/vertical): механически это className, но семантически это настройка-режим → Вариант с явной оговоркой.
- label и required-МАРКЕР (звёздочка) — field-level настройки состояния → Вариант, ЕДИНООБРАЗНО для всех контролов, даже когда их рисует не сам контрол, а обёртка FormField/makeFieldVariant. Разделять: наличие label и required-маркера = Вариант; enforcement обязательности валидатором = Пример.
- Подход к сборке (headless ручная композиция vs готовая ui-kit-обёртка, свободный layout) → Пример, НЕ Вариант. Как эталон допустимо оставить один baseline-Вариант дефолтного рендера, но акцент «разметку/стили пишете сами» и «готовая обёртка одной строкой» — это возможности.

### Несогласованности между компонентами

- **Одну и ту же opt-in булеву фичу-переключатель (добавляет affordance к собранному контролу) разные компоненты классифицировали противоположно: showToggle у InputPassword вынесен в Варианты ('С переключателем' / 'Без переключателя'), тогда как аналогичный clearable у Select признан возможностью и вынесен в Примеры. Оба — opt-in boolean, добавляющий иконку-действие.** _(InputPassword, Select)_ — Привести к правилу рубрики #8: opt-in фича = Пример. clearable у Select уже верно в Examples — оставить. У InputPassword сам reveal-приём вынести в Example ('Переключение видимости пароля', уже есть), а showToggle=false как РЕЖИМ собранного компонента ('поле повторите пароль', без кнопки) допустимо оставить одним Вариантом-состоянием. То есть: конфигурация-режим → 1 Вариант, интерактивная возможность reveal → Example. Не плодить два Варианта вокруг самой фичи.
- **label, приходящий из обёртки FormField/makeFieldVariant (а не собственный prop контрола), классифицирован по-разному: Input, Textarea, Select включают его в settingsAxes как ось Варианта наличия подписи; InputMask ЯВНО исключает label из settingsAxes ('не проп InputMask'); RadioGroup трактует FormField-label как Пример (FormField wiring). Checkbox имеет собственный prop label (отдельный случай).** _(Input, Textarea, Select, InputMask, RadioGroup, Checkbox)_ — Единое правило (рубрика #10): наличие label — field-level настройка → Вариант ВЕЗДЕ, независимо от того, кто её рисует (сам контрол или FormField-обёртка). Добавить label как ось Варианта в InputMask; в RadioGroup показ label оставить в Examples только в части aria-wiring, а наличие подписи учитывать как Вариант. Механизм (обёртка) фиксировать примечанием, но не менять классификацию.
- **required и required-маркер классифицированы непоследовательно: Textarea ('Обязательное'), Select ('Обязательный' через aria-required), FormField ('Required') держат required как Вариант-состояние; Checkbox и RadioGroup относят обязательность ТОЛЬКО к Примерам (валидация), не заводя Вариант, мотивируя тем, что маркер рисует FormField, а не контрол.** _(Textarea, Select, FormField, Checkbox, RadioGroup)_ — Разделить два смысла (рубрика #6, #10): (1) required-МАРКЕР/aria-required на собранном поле → Вариант ВЕЗДЕ, включая Checkbox и RadioGroup; (2) enforcement через validators: [required()] → Пример ВЕЗДЕ. Сейчас Checkbox/RadioGroup смешивают их в один Example — надо выделить Вариант 'Обязательное' и оставить Example 'валидация required'.
- **Runtime async/pending/loading-состояния отнесены то к Вариантам, то к Примерам. FormWizard заводит isValidating/isSubmitting как Варианты ('Загрузка: Проверка/Отправка'); AsyncBoundary заводит status=loading/error как Варианты; LoadingState — весь Вариант. Но FormField трактует pending (async-валидация) как возможность → Example; Select относит resource error/retry и загрузку к Examples.** _(FormWizard, AsyncBoundary, FormField, Select, LoadingState)_ — Единое правило: визуально различимое async-СОСТОЯНИЕ собранного компонента (спиннер, 'Проверка...', 'Отправка...', loading/error-экран) → Вариант; ПОДКЛЮЧЕНИЕ источника/цикла (fetch→status, resource-стратегии, чтение pending из контекста) → Пример. По этому правилу pending у FormField должен получить Вариант-состояние 'Pending' (сейчас только Example), а Select loading/error как визуальные состояния можно продублировать Вариантом, оставив стратегии resource в Examples.
- **«Ориентация через className» узаконена как Вариант в RadioGroup (единственное разрешённое исключение из запрета className-осей), но это создаёт риск прецедента: другой аналитик может по аналогии протащить в Варианты любую className-раскладку (grid-cols-2 у Section/Box, layout у FormField).** _(RadioGroup, Section, Box, FormField)_ — Зафиксировать в рубрике жёсткую границу: ЕДИНСТВЕННОЕ className-исключение — бинарная семантическая ориентация контрола (horizontal/vertical) с осмысленными состояниями. Любой layout-className (grid/flex/gap/space-y) остаётся стилизацией → Examples как приём, НЕ Вариант. Section (двухколоночная сетка) и Box уже верно держат layout в Examples/API — не переносить в Варианты.
- **Подход к сборке (headless ручная композиция и готовая ui-kit-обёртка) местами попал в Варианты. FormArray держит 'headless' (Root+List+AddButton) и 'styled' (FormArraySection) как Варианты; FormField держит 'Кастомная раскладка' (свободный layout) как Вариант. Оба анализа сами признают это ошибкой.** _(FormArray, FormField)_ — Правило рубрики #11: approach/composition/custom-layout → Example. Оставить ровно один baseline-Вариант дефолтного рендера ('Заполнен' у FormArray, 'Auto-render' у FormField), а 'headless-сборка', 'ui-kit-обёртка', 'свободный layout' перенести в Examples. FormArray/FormField уже пометили это в reclassification — применить.
- **Гранулярность декомпозиции витрины разная: Input/Textarea/Select/FormArray дают 8-11 Вариантов (мелкие состояния), тогда как InputPassword (5), RadioGroup (6), FormWizard (6) и особенно Collapsible (2), Section (3), LoadingState (3) — заметно грубее. Часть тонких компонентов при этом честно тонкие, но часть (InputPassword, RadioGroup) недобирает состояния (нет invalid/disabled/filled как отдельных карточек).** _(InputPassword, RadioGroup, Textarea, Input, Select, FormArray)_ — Договориться о минимальном каноне состояний-Вариантов для form-bound контролов: {пусто, заполнено, disabled, invalid, required, (+ типы/режимы, если есть)}. Тонкие presentational-компоненты (Collapsible/Section/LoadingState/ErrorState/Box) — исключение, у них состояний объективно мало. Довести InputPassword и RadioGroup до канона.
- **Native maxLength (Textarea) отнесён к Вариантам ('С нативным лимитом'), а бизнес-maxLength-валидатор — к Примерам. Это корректное разведение, но нигде больше native-ограничивающие атрибуты (min/max у Input type=number) не подняты в Варианты — Input держит min/max только в settingsAxes без карточки, а зажим к нулю — Пример.** _(Textarea, Input)_ — Единообразие: native hard-ограничения (maxLength, min/max), меняющие поведение ввода, — это настройка-режим → могут быть Вариантом ('с лимитом'/'число с min=0'), а активное поведение (зажим отрицательных, живой счётчик) — Пример. Либо оба в Вариантах, либо оба только в settingsAxes — сейчас Textarea и Input решили по-разному.

### Общие заметки

- Сквозной паттерн во ВСЕХ 17 анализах: почти каждый компонент в reclassification признаёт, что текущая doc-конфигурация недоукомплектована Вариантами-состояниями (нет invalid/disabled/filled/required карточек, хотя оси есть в settingsAxes) и Примерами-возможностями. Это не ошибка классификации, а пробел покрытия — при переработке страниц основной объём работы = ДОБАВИТЬ недостающие карточки, а не переклассифицировать существующие.
- Главный источник рассинхрона — обёртка FormField/makeFieldVariant. label, required-маркер, aria-invalid, aria-required технически рисует обёртка, а не сам контрол, и аналитики по-разному решали, считать ли это осью самого компонента. Нужно ЕДИНОЕ мета-правило: витринно контрол в доках всегда показан через FormField, поэтому field-level настройки (label, required, invalid) считаются Вариантами компонента независимо от того, чей это технически prop. Зафиксировать один раз и применить ко всем form-bound компонентам.
- Три презентационных/layout-компонента (Box fitsModel=false; Section и Collapsible — тонкие) ломают симметрию модели Variants/Examples. Для них страница должна деградировать корректно: у Box — только API/className-паттерны (ни Вариантов, ни Примеров), у Section/Collapsible — тонкая витрина (2-3 состояния) + Примеры-приёмы. Не пытаться раздувать их до размера Input/Select.
- Headless CDK-компоненты (FormField, FormArray, FormWizard) имеют перекос в сторону Examples по своей природе: почти вся поверхность — это возможности (compound-композиция, render-props, hooks, imperative ref, asChild). Их Variants — это витрина СОСТОЯНИЙ (пусто/заполнено/disabled/invalid/навигационная позиция), реализуемая преимущественно через ui-kit-обёртку. Это ожидаемо и не является дефектом — но требует явной оговорки на странице, чтобы читатель не искал 'настройки' там, где их мало.
- Риск при переработке: соблазн протащить стилизацию в Варианты под видом 'настройки раскладки'. Единственный узаконенный прецедент — RadioGroup orientation. Его надо огородить жёсткой формулировкой в рубрике, иначе grid-cols/flex/gap у Section, Box, FormField, FormArray.List поползут в Варианты и размоют границу styling→exclude.
- onChange/onBlur/binding везде классифицированы одинаково верно (wiring, не Вариант) — это самая консистентная ось, её можно использовать как эталон формулировки для рубрики. Аналогично disabled и invalid как Варианты — консистентны почти везде и служат якорем.
- Async/loading-состояния — второй по величине источник рассинхрона после FormField-обёртки: FormWizard/AsyncBoundary/LoadingState держат их в Вариантах, FormField/Select — в Примерах. Нужна явная развилка 'визуальное состояние (Вариант) vs подключение источника (Пример)', иначе страницы разных компонентов будут по-разному отвечать на вопрос 'как показать загрузку'.

**Переклассификация относительно текущего doc-конфига:**

- Вариант 'Email + валидация' сейчас смешивает НАСТРОЙКУ (type=email — корректный Variant) с ВОЗМОЖНОСТЬЮ (validators: required+email). По новому определению валидацию надо вынести в Examples, а в Варианте оставить чистый type=email без валидаторов.
- Вариант 'text' (Текст) — настройка, остаётся Variant. Ок.

## Box  —  `@reformer/ui-kit`

> ⚠️ Модель Variants/Examples применима слабо: Box — это чисто-layout/стилизационный компонент: `<div>` с ровно двумя props. `className` — единственная реальная «настройка», и она полностью стилизационная (layout через atomic CSS: flex-col, grid-cols-2, gap-4). `children` — не настройка и не возможность, а слот контента. У Box НЕТ ни одной не-стилизационной настройки (нет disabled/invalid/type/label/ориентации/required и т.п.) и нет ни одной opt-in возможности (нет clearable, resource, asChild, валидации, computed-поведения — компонент даже не пробрасывает произвольные HTML-атрибуты). Всё, что различает применения Box (вертикальный список vs двухколоночная сетка), задаётся исключительно через className, а стилизация по определению исключается из Вариантов. Поэтому модель Variants/Examples к Box неприменима: variants и examples пустые. Единственный осмысленный контент doc-страницы — таб API/использование, показывающий className-паттерны layout как стилизационные заготовки, но это не Варианты и не Примеры в смысле определения.
