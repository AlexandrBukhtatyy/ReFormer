# Стресс-тест контракта схемы поведения — «формы любой сложности»

## Context
Контракт `@reformer/core/behaviors` ([src/behaviors.ts](../../packages/reformer/src/behaviors.ts)) внедрён и проверен на кредитной форме. Теперь задача — **экспериментально подтвердить**, что им можно описать формы любого уровня сложности: выдумать реалистичные кейсы и тут же их протестировать. Тесты пишем как vitest-сценарии (быстро, детерминированно, прогоняются сразу), каждый сценарий = реалистичная форма + схема поведения + проверки. Где контракт упирается — документируем и закрываем расширением.

Известный гэп (из исследования): per-item поведение динамических массивов (`lineTotal = qty*price` на строку) и `apply` к элементам массива не работают — элементы-группы не зарегистрированы в реестре сигнал→нода. Агрегация по массиву (`compute` с map/reduce) — работает.

## Подход
1. Новый файл сценариев `packages/reformer/tests/behaviors/scenarios.test.ts` — ~18 кейсов по тирам сложности (ниже). Прогон итеративно, фиксируем pass/gap.
2. Закрыть per-item гэп маленьким оператором **`applyEach(arraySignal, itemSubSchema)`** в фасаде: реагирует на длину массива, (пере)применяет под-схему к каждой строке с per-item lifecycle (cleanup при remove). Это делает контракт пригодным для табличных форм любой сложности.
3. Спот-чек: существующие 3 формы (`/examples/complex*`) и полный набор тестов ядра не сломаны.

## Каталог сценариев (от простого к сложному)

**Тир 1 — вычисления и кросс-поле**
- S1 Заказ: `subtotal→tax(rate по региону)→discount(порог)→total` (цепочка compute + `{when}`).
- S2 Идемпотентный self-transform: `transformValue` (uppercase кода, digits-only телефон, clamp числа в диапазон).

**Тир 2 — условные поля**
- S3 Способ оплаты: card-поля enable при `card` (+resetOnDisable), bank-поля при `bank` (multi-target).
- S4 Каскад условий: поле C активно только когда A и B заполнены.

**Тир 3 — копирование/синхронизация**
- S5 Billing = Shipping (групповой `copyFrom` по флагу).
- S6 Двусторонняя `syncFields` (зеркало + transform: метры↔см).
- S7 `resetWhen` (сброс поля при смене условия) + `revalidateWhen` (ревалидация зависимого).

**Тир 4 — асинхронные зависимые справочники**
- S8 Каскад country→region→city (цепочка `onChange` + debounce; сброс нижестоящих; проверка отсутствия гонок/циклов).

**Тир 5 — массивы (агрегация — работает)**
- S9 Инвойс: `orderTotal = Σ(qty*price)` реактивно на add/remove/правку строки.
- S10 Условная секция по длине: `enableWhen(section, () => items.length > 0)`; агрегаты count/hasOverLimit.

**Тир 6 — массивы (per-item — закрываем гэп через `applyEach`)**
- S11 Per-row `lineTotal = qty*price` для каждой строки через `applyEach`; корректность при add/remove (новые строки получают поведение, удалённые — отписываются).
- S12 Per-row + агрегат вместе: строки считают свои lineTotal, родитель суммирует.

**Тир 7 — переиспользование/вложенность**
- S13 `apply` под-схемы адреса к двум адресам; вложенность 2 уровня (company.address + person.address).
- S14 Под-схема с пользовательским оператором внутри.

**Тир 8 — пользовательские операторы (открытость контракта)**
- S15 Авторинг доменных операторов: `requiredWhen`, `clampRange`, `maskDigits`, `mirrorWith` — композиция готовых; неотличимы от встроенных на месте вызова.
- S16 Низкоуровневый оператор через `effect`/`onDispose` (внешний источник/интервал) + проверка отписки.

**Тир 9 — жизненный цикл и guard**
- S17 `form.dispose()` останавливает все эффекты (никаких пересчётов после).
- S18 Dev-guard: оператор вне `defineFormBehavior` бросает понятную ошибку.

## Расширение для гэпа — `applyEach`
Файл [src/behaviors.ts](../../packages/reformer/src/behaviors.ts). Сигнатура: `applyEach<TItem>(array: ModelSignals<TItem[]> | object, itemSchema: FormBehavior<TItem>): void`.
- Реагирует на `array.length` (через `effect` на сигнал длины массива из `model.$`).
- Для текущего набора индексов строит per-item scope `{ model: <под-модель строки>, form: <под-форма строки> }` и запускает `itemSchema.__run(...)`, собирая cleanup на строку.
- При изменении длины: отписывает поведение исчезнувших строк, доприменяет к новым (по идентичности под-модели/ноды).
- Доступ к под-модели строки: `model.items.at(i)` (FormModel строки) уже есть; к под-форме строки: `form.items.at(i)` (ModelArrayNode.at). Если родительская форма не материализует массив-ноду — фолбэк на per-item форму из под-модели.

Если корректный per-item lifecycle окажется слишком объёмным — фиксируем гэп честно и даём паттерн-workaround в тесте (ручной re-apply на изменение длины), без расширения ядра.
