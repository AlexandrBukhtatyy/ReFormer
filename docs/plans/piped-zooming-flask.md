# TailwindCSS для layout — talking points для архитектурного комитета

## Context

Команда обсуждает внедрение TailwindCSS. Скоуп **двойной**:

- **Только layout** — позиционирование, отступы, размеры, flex/grid, alignment. Стили компонентов (кнопки, инпуты, карточки, типографика) остаются в UI-kit, Tailwind их не трогает.
- **Только новые микрофронты и shell** — legacy MFE остаются нетронутыми и **визуально не должны измениться ни на пиксель**. Это hard constraint.

Архитектура — Module Federation / single-spa: общий DOM, на одной странице сосуществуют legacy MFE и новые MFE. Главный технический риск — утечка CSS между MFE. Без решения по изоляции — внедрение блокировать **правильно**.

Цель этого документа — дать защищающему набор тезисов и заготовленные ответы. Это **не RFC**, не план миграции — это переговорная карточка.

---

## Talking points (5-минутный pitch)

### 1. Проблема, которую решаем

- Layout-код сегодня размазан: часть в CSS Modules, часть inline-стили (`style={{ marginTop: 16 }}`), часть — обёртки-«spacer»-компоненты, часть — ad-hoc классы в global CSS.
- Дизайн-токены spacing'а (4/8/12/16/24/32...) есть в дизайн-системе, но в коде воспроизводятся через магические числа. Консистентность держится на ревью, а не на инструменте.
- Простая правка «сдвинь блок на 8px» сейчас занимает: открыть css-файл, найти правильный класс, проверить что его не использует кто-то ещё, поправить, перезапустить HMR. На Tailwind — поправить класс в JSX.
- Для нового сценария или странички приходится либо плодить мелкие css-файлы под одноразовые layout'ы, либо писать inline-стили. И то, и другое — мусор.

### 2. Предложение

Подключить TailwindCSS как **layout-only** инструмент с **жёсткой изоляцией от legacy** через четыре независимых барьера:

**Что включаем (whitelist)**:
- `flex`, `grid`, `gap-*`, `items-*`, `justify-*`, `place-*`
- `p-*`, `px-*`, `py-*`, `m-*`, `mx-*`, `my-*`, `space-x-*`, `space-y-*`
- `w-*`, `h-*`, `min-w-*`, `max-w-*`, `min-h-*`, `max-h-*`
- `block`, `inline-*`, `hidden`
- `relative`, `absolute`, `fixed`, `sticky`, `top-*`, `left-*`, `right-*`, `bottom-*`, `z-*`, `inset-*`
- responsive-префиксы: `sm:`, `md:`, `lg:`, `xl:`

**Что НЕ подключаем** (отключаем через `corePlugins: { ... : false }`): цвета, типографика, скругления, тени, эффекты, бордеры (color). Всё это область UI-kit.

**Spacing scale** конфигурируем под наши design tokens (`tailwind.config.ts` → `theme.spacing`). Один источник правды для расстояний.

**Изоляция (см. отдельный раздел ниже)** — четыре барьера: prefix, layers, preflight off, scope selector.

### 3. Почему именно Tailwind (а не CSS Modules / styled-components / Panda / vanilla-extract)

- **Скорость DX**: классы прямо в JSX, без переключения файлов, без придумывания имён классов.
- **Покрытие сценариев**: responsive, dark mode, state-варианты (`hover:`, `focus:`) — всё из коробки. На CSS Modules каждый раз пишется руками.
- **Zero-runtime**: JIT генерирует ровно использованные классы в build-time, runtime-overhead = 0. styled-components/emotion дают runtime cost.
- **Tooling-зрелость**: VSCode-плагин с автокомплитом, ESLint-plugin для сортировки и валидации, Prettier-plugin. Команда не строит инструментарий с нуля.
- **Документация и экосистема**: для любого layout-вопроса есть ответ на SO/в доке за 30 секунд. Снижает bus factor.

### 4. Границы (важно проговорить явно)

**Компонентные границы**:
- UI-kit-компоненты **не используют Tailwind внутри себя** — там остаётся текущая стилизация.
- Tailwind применяется **снаружи** UI-kit — на уровне страниц, layout-секций, продуктовых композиций.
- Если для UI-kit-компонента нужен внешний margin/spacing — задаётся снаружи через Tailwind, **не** prop'ом компонента.
- ESLint-правило: запретить `style={{ ... }}` с layout-свойствами (`margin`, `padding`, `display`, `flex*`, `grid*` и т.д.) — форсирует единый канал для layout.

**MFE-границы (критично)**:
- Tailwind применяется **только в новых MFE и в shell**. Legacy MFE **не трогаем**.
- Внутри shell — только layout-контейнеров, в которых рендерятся MFE-слоты. Содержимое legacy MFE остаётся под своей CSS-крышкой.
- Legacy MFE **не теряют** контроль над своими стилями: четыре барьера изоляции (см. ниже) гарантируют, что наш Tailwind физически не дотягивается до их DOM.

### 4a. MFE-изоляция: четыре барьера

Каждый барьер закрывает свой класс утечки. Они независимы — даже если один обойдёт, остальные остановят.

**Барьер 1 — Class prefix (`prefix: 'tw-'` в `tailwind.config.ts`)**
- Все классы получают префикс: `tw-flex`, `tw-p-4`, `tw-gap-3`.
- Исключает коллизии имён с глобальными CSS legacy MFE. Если у них где-то есть свой `.flex {...}` — наш `.tw-flex` его не заденет, и наоборот.
- Стоимость: чуть длиннее имена классов. Окупается полностью.

**Барьер 2 — Preflight off (`corePlugins: { preflight: false }`)**
- Tailwind по умолчанию инжектит CSS reset (Preflight), который применяется глобально на `*`, `html`, `body`, `h1-h6`, `button`, `a` и т.д. **На общем DOM с legacy MFE это категорически нельзя** — гарантированно поломает их типографику и формы.
- Полностью отключаем. Reset — забота того, у кого он уже есть.

**Барьер 3 — CSS Cascade Layers (`@layer`)**
- Все наши Tailwind-стили оборачиваются в `@layer tailwind-layout { ... }`.
- По спецификации CSS Cascade Layers: правила в layer всегда **проигрывают** unlayered-правилам, **независимо от специфичности**.
- Legacy CSS (без `@layer`) — unlayered → перебивает наш Tailwind. То есть **наш Tailwind физически не может перетереть стили legacy MFE**, даже случайно. Это самый важный барьер.
- Поддержка: все современные браузеры с 2022 года (Chrome 99+, Firefox 97+, Safari 15.4+).

**Барьер 4 — Scope selector (`important: '#new-mfe-root'`)**
- Tailwind поддерживает scoping через `important: <selector>` — все классы превращаются в `#new-mfe-root .tw-flex { ... }`.
- Гарантирует, что Tailwind **работает только внутри корневого DOM нашего MFE**, и не задевает соседей.
- Внутри своего scope — наши классы перебивают пришедшие извне стили (благодаря `important`-селектору с высокой специфичностью), что нам и нужно: внутри новых MFE Tailwind работает предсказуемо.

**Итог**: legacy MFE **не пострадает**. Это проверяется автоматическим визуальным регресс-тестом на shell-странице (см. план внедрения).

### 5. План внедрения (низкий риск, MFE-aware)

1. **Isolation PoC** (2-3 дня) — **первый и обязательный шаг**. Включаем Tailwind в одном новом MFE со всеми четырьмя барьерами. На тестовой shell-странице рядом ставим **самый сложный legacy MFE**. Запускаем визуальную регрессию (Playwright + screenshot diff) до и после. **Acceptance criteria**: 0 пикселей разницы в legacy MFE. Если хоть что-то ломается — возвращаемся к комитету с разбором, без расширения.
2. **Shell layout** (1 неделя) — после прохождения PoC переводим layout-контейнеры shell на Tailwind. Это даёт единый язык для расположения MFE-слотов. Регрессия на legacy MFE — повторяем.
3. **Pilot-модуль в новом MFE** (1 неделя) — один продуктовый MFE полностью на Tailwind для layout. Собираем фидбек команды, замеряем bundle, время правки.
4. **Расширение по новым MFE** (next) — все новые MFE пишутся с Tailwind по тому же шаблону config'а. Legacy MFE не трогаем никогда.
5. **Lint-gate** — после pilot'а включаем ESLint-правила, блокирующие новые layout-стили вне Tailwind в новых MFE. На legacy lint не распространяется.

### 6. Метрики успеха (что покажем через квартал)

- **Регрессии в legacy MFE: 0** (главный KPI — измеряется визуальной автоматикой на CI каждый PR в новых MFE/shell).
- Среднее время правки layout-задачи в новых MFE.
- Количество ad-hoc css-файлов с layout-only стилями в новых MFE (должно уменьшаться).
- Bundle size новых MFE до/после (ожидаем рост ~5-10kb gzip на MFE).
- NPS команды по DX (опросник).

## Q&A — заготовленные ответы

### Q: «А что с микрофронтами? Module Federation — общий DOM. Tailwind утечёт в legacy MFE и сломает их визуал. Мы не можем туда внести правки.»

Это главный вопрос, и ответ структурный — четыре независимых барьера (см. раздел «MFE-изоляция»), плюс автоматический визуальный регресс-тест.

**Короткая версия для слайда**:
1. **Prefix `tw-`** — нет коллизий имён классов с legacy.
2. **Preflight OFF** — нет глобального reset, который мог бы тронуть legacy DOM.
3. **CSS Cascade Layers** — по спеке наш layered Tailwind физически **не может перебить** unlayered legacy CSS, даже случайно.
4. **Scope selector** (`important: '#new-mfe-root'`) — наши классы работают **только внутри корня нашего MFE**.
5. **Visual regression на CI** — каждый PR в новых MFE/shell прогоняется через screenshot-diff на тестовой странице с самым сложным legacy MFE. Acceptance: **0 пикселей разницы**.

**Почему этого достаточно**: чтобы Tailwind сломал legacy, ему пришлось бы (а) совпасть по имени класса И (б) обойти cascade layers И (в) применить стиль вне scope-селектора. Все три одновременно невозможно — каждый барьер достаточен сам по себе.

**Что мы делаем, если PoC ломает legacy**: возвращаемся к комитету с разбором, расширение не идёт. Это hard gate в плане, не «потом разберёмся».

### Q: «А shell? Если shell на Tailwind, его layout-классы будут рядом с DOM legacy MFE — точно ничего не сломается?»

- Shell использует Tailwind **только для layout-родителей**, в которые рендерятся MFE-слоты. Внутри слота — DOM legacy MFE с его собственными стилями.
- Tailwind-классы на shell-уровне (`tw-flex tw-gap-4`) применяются к **родительскому** div'у. Они влияют только на расположение **самого** слота на странице, не на содержимое legacy MFE.
- Layout-grid (где какой MFE стоит) — это **ответственность shell всегда**, независимо от Tailwind. Сегодня это делается через css-grid в собственном CSS shell'а. Tailwind просто заменяет «как написано», не «кто что решает».

### Q: «Bundle size? Это ещё одна зависимость, и Tailwind генерирует мегабайты CSS.»

- Tailwind с JIT-режимом (с v3 — дефолт) генерирует **только использованные классы**. Если в коде нет `flex`, в бандле нет `flex`.
- В нашей конфигурации (whitelist на layout-only) полный output даже **до purge** уже маленький — порядка 30-50kb несжатого. После gzip — **5-10kb на один MFE**.
- **Дублирование между MFE**: каждый MFE собирает свой Tailwind-CSS. Но классы детерминированы (`.tw-flex { display: flex }` определён одинаково везде), браузер кеширует одинаковые байты CSS неэффективно — но gzip и HTTP/2-мультиплексирование сводят реальные потери к единицам KB.
- Для сравнения: один шрифтовой файл — 30-100kb. CSS Modules с layout-классами на 50 страниц легко даёт сопоставимый объём, но **дублирующимися правилами** (каждый файл свои `display: flex`).
- Build-time overhead: +1-3 секунды на сборку. Незаметно на фоне TypeScript.
- **Зависимость**: один npm-пакет (`tailwindcss`), активно поддерживаемый, миллионы downloads/нед, нет рисков заброшенности.

### Q: «Это конфликтует с UI-kit. У нас уже своя дизайн-система.»

- **Не конфликтует, потому что не пересекается по скоупу.** UI-kit отвечает за компоненты (Button, Input, Card, Typography). Tailwind отвечает за **то, что находится между компонентами и вокруг них** — layout grid, отступы между секциями, выравнивание.
- Это две разные оси:
  - UI-kit: «как выглядит кнопка»
  - Tailwind: «где стоит кнопка на странице и какой у неё внешний отступ»
- Сегодня обе оси перемешаны: UI-kit-компоненты иногда принимают `margin` prop, иногда нет; layout-обёртки иногда написаны как UI-kit-компоненты (`<HStack>`, `<Spacer>`), иногда как inline CSS.
- Tailwind в layout-скоупе **проясняет** границу: компонент = UI-kit, расположение = Tailwind. Это упрощает архитектуру, а не усложняет.
- Дополнительно: spacing-scale Tailwind мы конфигурируем **из** design tokens. То есть Tailwind становится способом **применять** дизайн-токены к layout, а не альтернативной системой токенов.

### Q: «Utility-soup. JSX превратится в нечитаемые портянки `className="flex items-center justify-between p-4 gap-3 md:p-6 ..."`.»

- Это реальный риск **при неограниченном скоупе**. Мы его исключаем тремя механизмами:
  1. **Скоуп**: только layout-утилиты (~30 классов категорий). Не цвета, не типографика, не варианты. Длина className резко падает.
  2. **Линтер**: `eslint-plugin-tailwindcss` сортирует классы детерминированно (читать привыкаешь за неделю) и валидирует.
  3. **Когда длинно — выносим**: для повторяющихся layout-паттернов делаем CDK-компонент (`<PageSection>`, `<FormRow>`) или используем `clsx`/`cva` для именованных вариантов. Это абсолютно нормальный Tailwind-pattern.
- Альтернативы тоже имеют шум: CSS Modules дают шум **в виде десятков css-файлов и придумывания имён**; styled-components — **в виде десятков styled-обёрток-компонентов**. Шум есть всегда, вопрос — где он лучше виден и легче чинится.
- В JSX layout-шум **колокально с разметкой**, его видно при чтении. В отдельных файлах его не видно — пока кто-то не сломает.

### Q: «А что, если завтра захотим уйти с Tailwind?»

- Layout-only Tailwind — это формализованное применение CSS-utility-классов. Миграция = автоматическая замена `className="flex gap-4"` на соответствующие CSS-правила (можно скриптом). Это самый простой для отката outof-vendor-инструмент по сравнению со styled-components (runtime-зависимость в JS-коде) или Panda (compile-time-трансформация).
- Vendor lock-in минимальный, потому что мы не используем плагин-экосистему и кастомные API.

### Q: «Команда не знает Tailwind. Обучение, ошибки, замедление.»

- Layout-скоуп — это ~30 классов с очевидной семантикой (`flex`, `gap-4`, `p-6`, `items-center`). За полдня осваивается полностью.
- VSCode-плагин даёт автокомплит с превью значений в пикселях — учиться по ходу, а не заранее.
- Документация Tailwind — одна из лучших в индустрии, бесплатная, с поиском.
- Risk-mitigation: pilot-модуль ведут 1-2 senior'а, остальные подхватывают по примерам.

### Q: «Почему сейчас, а не через год?»

- Чем дольше живём с разнородным layout-кодом, тем дороже миграция. Каждая новая страница увеличивает технический долг.
- Скоуп ограничен — внедрение **не блокирует** другие инициативы. Это аддитивное изменение, не реворк.
- Конкретный тригерр (заполнить по ситуации): новая фича X / редизайн Y / онбординг новых разработчиков Z.

### Q (запас): «Почему не `@emotion/styled-system` / `chakra-ui-styled-system` props-based подход?»

- Props-based spacing (`<Box mt={4} p={2}>`) требует **обёртки-компонента** для каждого DOM-узла. Это рантайм-цена и шум в дереве React DevTools.
- Tailwind: те же возможности, **ноль рантайма**, любые html-теги (`<section>`, `<header>`, `<main>`) без оборачивания.

### Q (запас): «А responsive? У нас десктоп-only / мобильный-only.»

- Если только desktop — отключаем responsive-префиксы в конфиге. Один параметр.
- Если в будущем понадобится — он уже встроен, не нужна вторая миграция.

## Что не входит в это предложение (явно)

- **Любые изменения в legacy MFE** — не трогаем ни код, ни стили, ни build.
- Переписывание UI-kit на Tailwind.
- Использование Tailwind для цветов, типографики, теней, бордеров, скруглений.
- Plugin-экосистема Tailwind (typography plugin, forms plugin) — не включаем.
- Замена существующих CSS Modules в UI-kit.
- Большая миграция «всё или ничего».
- Tailwind Preflight (CSS reset) — категорически выключен из-за общего DOM.
